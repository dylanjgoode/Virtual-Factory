MQTT Glossary

i

Quick reference for the core MQTT concepts used in this dashboard.

Topic

A text path that labels a message, like factory/telemetry/press/temperature.

Wildcards

Use + for one level and # for all remaining levels in a subscription.

QoS

Quality of Service. QoS 0 is best-effort, QoS 1 retries until acknowledged.

Retained

Retained messages are stored by the broker and replayed to new subscribers.

LWT

Last Will and Testament. A broker-published offline status when a client dies.

Sessions + ACLs

Persistent sessions queue QoS 1 messages. ACLs restrict who can publish/subscribe.

MQTT In Motion

Devices publish telemetry, alarms, state, and status into the broker. The broker fans out those topics to the dashboard, controller, and any observer while caching retained state for instant onboarding.

Publish

Devices stream telemetry (QoS 0) and alarms (QoS 1) into topic lanes like telemetry/{device}/{sensor} and alarms/{device}/{sensor}.

Retain + LWT

State and status are retained by the broker; LWT flips a status offline when a client disappears uncleanly.

Fan-out

Controller and dashboard subscribe broadly, so the broker can replay retained state and deliver new alarms instantly.

Commands

Dashboard + controller publish commands (QoS 1) to commands/{sender}/{device}; the broker delivers them to devices, which answer on state/status.

Why instrument telemetry

  • Detect drift before it becomes downtime.
  • Correlate alarms with the data that caused them.
  • Track performance so you can tune the line.
  • Build trust with a clear record of what happened.
Telemetry shows what is happening now; retained state shows what should be true for new subscribers.

Why MQTT here

  • Publish/subscribe keeps machines and tools loosely coupled.
  • QoS 0 for fast telemetry, QoS 1 for critical alarms and commands.
  • Retained messages give instant context when you connect.
  • Last Will makes device presence automatic.
